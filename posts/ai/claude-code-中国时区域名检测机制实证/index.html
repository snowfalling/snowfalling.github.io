Claude Code 中国时区/域名检测机制实证#

对 Claude Code CLI 二进制中一段隐蔽逻辑的分析记录。该逻辑基于系统时区、ANTHROPIC_BASE_URL 域名、以及中国 AI Lab 关键字，改变注入到模型 system prompt 中的日期格式与撇号字符。

  • 环境：macOS Darwin 25.4.0，Asia/Shanghai
  • Claude Code 版本：2.1.197（~/.local/share/claude/versions/2.1.197，Mach-O arm64）
  • 记录日期：2026-07-01

一、背景#

偶然刷到公众号，说是claude有针对中国区的特殊验证逻辑，后来在一些网站上看了下确有此事。近几天开发编程工作用claude还是蛮顺手的，所以试着想想看能否找到claude的相关代码

二、与claude的对话沟通#

对claude进行提问

检查系统时区是否匹配 Asia/Shanghai 或 Asia/Urumqi。

检查你的代理 URL 是否是中国域名，是否匹配某个域名列表，或者是否包含某个中国 AI Lab。

基于上述两项检查，Anthropic 会修改 system prompt 中的日期部分。

如果系统时区被判断为中国，日期格式会从 2026-06-30 变成 2026/06/30。同时，根据代理 URL 的不同，Today’s date is 中的撇号也会发生变化：

如果是中国域名，或者匹配域名白名单，但不是 AI Lab：使用 \u2019，即 “right single quotation mark”——’

如果不是中国域名，或者匹配域名白名单，但是中国 AI Lab：使用 \u02BC，即 “modifier letter apostrophe”——ʼ

如果是中国域名，或者匹配域名白名单，且同时是中国 AI Lab：使用 \u02B9，即 “modifier letter prime”——ʹ

验证下以上内容是否真实

function udp() {                                    // 读代理 host
  let e = process.env.ANTHROPIC_BASE_URL;
  if (!e) return null;
  try { return new URL(e).hostname.toLowerCase() } catch { return null }
}

function ddp() {                                    // 三项检测
  if (Crt()) return null;
  let e = udp();
  let t = e0t();                                    // 系统时区
  let n = t === "Asia/Shanghai" || t === "Asia/Urumqi";
  if (!e) return { known: false, labKw: false, cnTZ: n, host: null };
  return {
    known: ldp().some(r => e === r || e.endsWith("." + r)),   // 域名白名单
    labKw: cdp().some(r => e.includes(r)),                    // AI Lab 关键字
    cnTZ:  n,
    host:  e
  };
}

function pdp(known, labKw) {                        // 选择撇号
  if (!known && !labKw) return "'";        // U+0027
  if ( known && !labKw) return "’";   // ’
  if (!known &&  labKw) return "ʼ";   // ʼ
  return "ʹ";                         // ʹ
}

function eca(e) {                                   // 组装 "Today's date is ..."
  let t = ddp();
  let n = pdp(t?.known ?? false, t?.labKw ?? false);
  let r = t?.cnTZ ? e.replaceAll("-", "/") : e;     // 中国时区 → 斜杠日期
  return `Today${n}s date is ${r}.`;
}

function Zla(e){
  let t = Buffer.from(e,"base64"), n = "";
  for (let r of t) n += String.fromCharCode(r ^ 91);   // sdp = 91
  return n.split(",");
}

五、对本机的具体判定#

  • 时区：Asia/Shanghai → cnTZ = true → 注入日期改写为 2026/07/01（斜杠）
  • ANTHROPIC_BASE_URL：未设置 → known = false, labKw = false → 撇号为普通 U+0027

预测结果：Today's date is 2026/07/01.

实际观察到的 system prompt currentDate 段落与该预测完全一致。

六、结论#

  • 该逻辑确实存在于 Claude Code CLI 二进制中，源码级证据可复现。
  • 检测三项：系统时区（Asia/Shanghai / Asia/Urumqi）、ANTHROPIC_BASE_URL 主机名对 147 项域名白名单、对 11 项 AI Lab 关键字。
  • 输出信号：日期分隔符（- vs /）+ 四种撇号字符（' / ’ / ʼ / ʹ）—— 编码空间足以表达 2×4 = 8 种组合。
  • 白名单与关键字通过 base64+XOR 混淆，非明文；strings 直接搜索无法命中列表内容，只有 Asia/Shanghai / Asia/Urumqi 明文暴露。
  • 官方文档中未见对该机制的说明。用途从代码本身无法确定（可能用于上游区分/统计/降级/审计等）。

BIN=$(readlink $(which claude))
echo "Binary: $BIN"

# Step A: 明文标记（Asia/Shanghai、Asia/Urumqi、Today's date）
strings "$BIN" | grep -nE "Asia/Shanghai|Asia/Urumqi|Today.{0,8}s date"

# Step B: 提取 udp / ddp / pdp / eca 四个函数的完整源码
python3 - "$BIN" <<'PYEOF'
import re, sys
data = open(sys.argv[1], "rb").read()
m = re.search(rb'function udp\(\)\{let e=process\.env\.ANTHROPIC_BASE_URL', data)
if not m:
    print("NOT FOUND"); sys.exit(1)
blob = data[m.start():m.start()+900].decode("utf-8", "replace")
for part in re.split(r'(?=function \w+\()', blob):
    if part.strip():
        print(part.rstrip(), "\n")
PYEOF

# Step C: 解码混淆的域名白名单 / AI Lab 关键字列表
python3 - "$BIN" <<'PYEOF'
import re, sys, base64
data = open(sys.argv[1], "rb").read()
m_idp = re.search(rb'idp="([A-Za-z0-9+/=]{500,})"', data)
m_adp = re.search(rb'adp="([A-Za-z0-9+/=]{80,})"',  data)
dec = lambda s: "".join(chr(x ^ 91) for x in base64.b64decode(s)).split(",")
if m_idp:
    lst = dec(m_idp.group(1))
    print(f"DOMAIN LIST ({len(lst)} entries):")
    for d in lst: print(f"  {d}")
if m_adp:
    lst = dec(m_adp.group(1))
    print(f"\nLAB KEYWORDS ({len(lst)} entries):")
    for d in lst: print(f"  {d}")
PYEOF

常见坑#

  1. Step A 的正则不能写成 Today.{0,3}s date。eca() 里的模板字符串是 `Today${n}s date is ${r}.`，编译后 Today 与 s date 之间被 ${n} 占位符隔开，长度超过 3，会 grep 不到，导致误判"二进制里没有这段逻辑"。至少要用 .{0,8}。
  2. Step B 用锚点 function udp(){let e=process.env.ANTHROPIC_BASE_URL 一次性定位。四个函数（udp / ddp / pdp / eca）在打包后是紧邻排列的一整块，从 udp 开头往后取 ~900 字节就能覆盖到 eca 结束。
  3. Step C 中的 base64 串必须从二进制中程序化提取，不要手工从终端复制——两个字符串都很长（idp 约 3KB，adp 约 130 字节），复制过程中极易掉字符导致 base64 解码失败。
  4. XOR key = 91 硬编码在 sdp=91，如果未来版本调整了这个值，用 re.search(rb'var sdp=(\d+)', data) 提取即可。

版本适用性#

上述脚本在 Claude Code v2.1.197 上验证通过。若日后版本变更导致命名混淆变化，用于定位的锚点字符串 process.env.ANTHROPIC_BASE_URL、Asia/Shanghai、Asia/Urumqi 大概率仍会保留在明文中，可以据此重新调整正则。
